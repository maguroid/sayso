# Fable 保守セッション ブリーフィングパック

2026-07-08 以降、Fable は従量課金になる。sayso の改善は不定期に頼むのではなく、**計画的な単発の Fable 保守セッション**（目安: 月1回・30分）として予算化する。このファイルはそのセッションの冒頭で読む唯一のエントリポイントであり、コンテキスト再構築のコストを最小化するために存在する。

## あなた（保守セッションの Fable）へ

あなたは sayso（Sonnet 5 を Fable 級オーケストレーターとして起動するコンテキストフレームワーク）の保守者である。このフレームワークはあなた自身の実セッション47件の行動分析から、あなた（の過去のセッション）が作った。

### 読む順序（これ以外を先に読まない — コンテキスト節約）

1. このファイル
2. `eval/protocol.md` — 評価と改訂の規律（特に §4 改訂プロトコル・規則予算制）
3. **バックログ**: `gh issue list --label backlog:framework --state open` — M4 実運用で見つかった問題の記録
4. 直近の `eval/judgments/`（新しい順に2〜3本で十分）
5. 必要時のみ: `framework/ORCHESTRATOR.md` 全文、非公開の実セッション行動分析ノート（作者手元、47セッション分。行動規範の原典）

### セッションの定型フロー

1. バックログの各 issue を読み、乖離を3分類する（規範の欠落／規範はあるが不従／地頭差）。
2. 「欠落」のみ規則追加の対象。**規則予算制を厳守**: ORCHESTRATOR.md は約250行上限。足すなら、効いていない規則を削るか統合する。同一テーマから連続で規則を導かない。
3. 改訂したら、関係する静的シナリオ（`eval/run.sh`）の回帰を回す。ライブ比較が必要なら**別角度のお題**で（お題バンクと出所隔離は protocol.md §2）。ゴールドは `eval/gold-scenarios/` の収録済み Fable 走行を使い、**新たな Fable 実行はしない**。
4. 成果物: (a) ORCHESTRATOR.md の改訂コミット（バージョンをコミットメッセージに明記）、(b) `eval/judgments/` への記録、(c) 処理した issue のクローズ（判断理由をコメント）、(d) `sayso update` の案内。

### 禁止事項（このセッションでも有効）

- **Fable の追加実行・並列実行をしない**（あなた自身のこのセッションが唯一の Fable）。ジャッジが要る場合は opus/codex サブエージェントを使う。
- 規則予算を超える肥大化をしない。「過度に規定的なプロンプトは品質を下げる」。
- バックログにない問題を探しに行かない（スコープはバックログ処理。新しい探索はユーザーが明示的に依頼した場合のみ）。

## ユーザー（maguroid）へ: バックログの書き方

M4 実運用で sayso の問題に気づいたら、その場で issue を切る（保守セッションの品質はバックログの再現性で決まる）:

```
gh issue create -R maguroid/sayso -l backlog:framework -t "<一言>" -b "
- 状況: <どんなタスクを投げたか。可能ならプロンプト原文>
- 期待: <Fableならどうしたはずか / どうしてほしかったか>
- 実際: <sayso が何をしたか。トランスクリプトの該当部があれば貼る>
"
```

期待と実際が書けない場合も「なんか違和感があった」だけで切ってよい（保守セッションが分類する）。
