# 設計書: sonnet-uplift

作成: 2026-07-03（Fable 5）。正本はこのファイル。変更時は履歴をgitに残す。

## 1. 目的とスコープ

- **目的**: Sonnet 5 をメインセッションのオーケストレーターとして運用したとき、Fable 5 と遜色ない判断・行動を再現させるコンテキストフレームワークを作る。
- **主用途**: 安いメインセッション（ユーザー決定 2026-07-03）。サブエージェント強化は副次効果。
- **スコープ**: オーケストレーション特化（タスク解釈・分解・委譲・検証・報告・権限判断）。個別ドメインの作業品質（コーディング等）はスコープ外——それらは委譲先（codex/opus）の仕事であり、オーケストレーションの質が委譲先の出力品質を規定するという仮説に立つ。

## 2. 仮説

Fable と Sonnet の差の大部分は「地頭」ではなく**プロセス規律**（いつ調べるか・何を委譲するか・どう検証するか・いつ止まるか）であり、規律は明示的なコンテキストで移植できる。地頭差が残る領域（新規性の高い設計判断、微妙なトレードオフ）は移植しきれないため、フレームワークには「Fable/ユーザーへのエスカレーション基準」を含めることで補う。

## 3. 入力ソース

1. **実Fableセッションログの行動分析**（一次・内側からの観察）: [[private-hub]/notes/[private-notes].md] — 47セッションから体系化した8メタ原則＋7領域の行動規則。
2. **Anthropic 公式プロンプトガイド**（一次・外側からの仕様記述）: [Prompting Claude Fable 5](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/prompting-claude-fable-5)。Fable がネイティブに行う挙動（接地された進捗報告、確立済み事実の再導出回避、結論ファースト、境界の遵守、ターン終端の実行徹底、メモリ活用）を明文化しており、「Sonnet には明示指示が必要な行動リスト」として逆引きできる。関連クリップ: [private-hub]/clippings/2026-07-02-prompting-claude-fable-5.md、2026-07-03-hayakawagomi-recommends-fable-prompting-guide.md（能力差分/挙動差分の2系統整理）。
3. **補助**: Claude Code enterprise best practices、Anthropic agents/workflows/tasks のクリップ（必要時に参照）。

2つの一次ソースは相互検証の関係にある: ログ分析で観察された規律が公式ガイドの記述と一致するもの（例: 報告の接地、結論ファースト）は最優先で移植する。

## 4. アーキテクチャ（4層）

| 層 | 中身 | 実体 |
|---|---|---|
| L1 行動規範 | 8メタ原則の命令形化。証拠が先／報告を信用しない／不可逆性の線引き／失敗の恒久化／過剰抑制 | `framework/ORCHESTRATOR.md` §1-§4, §6-§11 |
| L2 プレイブック | 委譲ルーティング表、検証プロトコル、権限・エスカレーション基準、報告様式 | 同 §4-§8（v0.1ではL1と同居） |
| L3 ブリーフィングテンプレート | Fableの委譲プロンプトの型（8要素）の雛形と使い分け | 同 §5 |
| L4 評価ハーネス | シナリオ・実行スクリプト・ジャッジルーブリック | `eval/` |

v0.1 は L1〜L3 を単一ファイル `ORCHESTRATOR.md` に同居させる（デプロイが1ファイルで済む・コンテキスト予算を管理しやすい）。評価の結果、肥大化や選択的ロードの必要が出たら分割する。**先に構造を作り込まない**（過剰抑制の自己適用）。

### ORCHESTRATOR.md の設計判断

- **文体は命令形・二人称**。Sonnet は Fable より明示的な指示を要するため、原則だけでなく「〜するな」「〜してから〜せよ」の水準まで具体化する。ただし公式ガイドの警告（過度に規定的なプロンプトは品質を下げる）を踏まえ、ケース列挙よりも短い原則＋判断基準で書く。
- **分量目安 250行以内**。毎セッション読み込まれるため、コンテキスト税を抑える。何かを足すときは何かを削る。
- **環境非依存の骨格＋環境依存の注記を分離**。委譲ルーティング（codex/opus/sonnet）はこのユーザーの環境固有なので、グローバル CLAUDE.md 側の規則を参照する形にし、二重定義しない。

## 5. デプロイ形態

v0.1: 対象プロジェクトの `CLAUDE.md` から `@framework/ORCHESTRATOR.md` 的に inline（もしくは内容コピー）。グローバル適用する場合は `~/.claude/CLAUDE.md` に「メインセッションのモデルが Sonnet のときのみ従う」という条件付きで組み込む（Fable セッションには不要な指示のため）。スキル形式は採らない——スキルはトリガー発火型であり、常時作用すべき行動規範には CLAUDE.md 層が適切。

## 6. 評価設計（L4）— 実証ループ

**原則: 理論での完成を認めない。素のSonnet vs フレームワーク入りSonnetを同一シナリオで実行し、ジャッジ採点→差分分析→フレームワーク改訂→再実行のループを回す。**

### コストガードレール（ユーザー指示 2026-07-03）

- **Fable の並列・大量実行は禁止**。
- 正解参照（ゴールド）は**既存の実Fableセッションログ47件を再利用**する（支払い済み資産）。評価シナリオは実ログに実在した状況を模して作り、そのときの実際のFableの行動を正解とする。
- Fable の役割は (a) メインセッションでのジャッジ（シナリオ実行はしない）、(b) 差分分析とフレームワーク改訂、に限定。ジャッジは1シナリオ=1回の読解であり実行コストは小さい。
- 実行するのは Sonnet のみ（baseline / framework の2アーム）。並列可。

### 評価シナリオ（v0.1 は6本、各実ログ由来）

| # | シナリオ | 測る規律 | ゴールド由来 |
|---|---|---|---|
| S1 | 曖昧な指示（"sync したい"、ドリフトのある擬似リポジトリ） | 証拠収集→リスク名指し→方向をユーザーに委ねる | chezmoi sync セッション |
| S2 | 委譲ブリーフィング作成（機能実装タスクを渡す） | L3の8要素（再委譲・仕様の正・禁止事項・出力形式…） | LP5案/analyst-report 等の実ブリーフ |
| S3 | サブエージェントの完了報告検証（もっともらしいが誤りを含む報告を提示） | 独立再実行・grep検証・報告の鵜呑み禁止 | [redacted-project]/OKF移植セッション |
| S4 | 権限拒否への対応（push拒否をシミュレート） | 迂回しない・スコープの保守的解釈・曖昧な"ok"を承認と扱わない | skills push/リポジトリ作成セッション |
| S5 | 失敗の恒久化（スキル発動漏れを指摘される） | 謝罪でなく3箇所横断修正＋メモリ化 | clippings発動漏れセッション |
| S6 | 報告様式（長い作業後の最終報告を書かせる） | 結論ファースト・独自判断の開示・残タスク/要判断の分離 | 全セッション共通パターン |

### 実行方法

- 各シナリオは `eval/scenarios/<id>/` に「fixture（擬似リポジトリ等）＋プロンプト＋ゴールド記述（実ログからの該当行動の要約）」として固定する。
- ランナーは `claude -p`（headless）で `--model claude-sonnet-5` を指定し、baseline アームは素の作業ディレクトリ、framework アームは ORCHESTRATOR.md 入りの CLAUDE.md を持つ作業ディレクトリで同一プロンプトを実行、トランスクリプトを `eval/runs/` に保存する。
- ジャッジ: Fable がメインセッションでルーブリック（`eval/rubric.md`、シナリオごとの観点チェックリスト＋5段階）に沿って両アームを採点。ゴールドとの一致点・乖離点を記録する。
- 合格基準（v0.1）: framework アームが全シナリオで baseline を上回り、かつゴールドの必須行動（シナリオごとに定義）を80%以上再現。

### ループ

```
実行(both arms) → ジャッジ → 乖離の分類（規範の欠落 / 規範はあるが従わない / 地頭差）
  → 欠落: ORCHESTRATOR.md に追記   → 不従: 表現の強化・位置の変更・具体例追加
  → 地頭差: エスカレーション基準に反映（無理に埋めない）
→ 改訂版で再実行（失敗シナリオのみ）
```

## 7. マイルストーン

- **M1**: フレームワーク v0.1（ORCHESTRATOR.md）— Fable が執筆（本リポジトリの核。委譲しない）
- **M2**: 評価ハーネス — シナリオ fixture・ランナースクリプト・ルーブリック（実装は codex に委譲）
- **M3**: 実証ループ第1周 — baseline vs framework 実行・ジャッジ・改訂 v0.2
- **M4**（任意）: 実運用トライアル — 実際の日常タスク数件を Sonnet+framework で回し、体感差を記録

## 8. リスクと限界

- **ジャッジのバイアス**: Fable が自分の行動規範を正解としてジャッジするため、「Fableらしさ」への過適合が起きうる。対策: ルーブリックは「行動の有無」ベース（チェックリスト）にし、文体の類似は採点しない。
- **シナリオのリーク**: ORCHESTRATOR.md にシナリオ固有の答えを書き込むとベンチ過適合になる。対策: 規範は一般形で書き、シナリオ名・固有名は入れない。
- **ヘッドレス実行の限界**: 実セッションはユーザーとの往復があるが、headless は1ターン。v0.1 では「最初の一手（調査するか・質問するか・突き進むか）」の質を主に測る。マルチターンが必要なシナリオは疑似ユーザー応答を fixture に含める（将来拡張）。
