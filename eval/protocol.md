# 評価プロトコル — ライブタスク比較とベンチ最適化の防止

2026-07-03 のフラット3アーム比較（eval/judgments/2026-07-03-flat3-design-consult.md）で確立した方法論の型。フレームワーク改訂の効果測定はこのプロトコルに従う。

## 1. 評価の3形態

| 形態 | 何を測る | 頻度・使いどころ |
|---|---|---|
| **静的シナリオ**（S1〜S7、`eval/run.sh`） | 特定の規律の有無（回帰テスト） | フレームワーク改訂のたび。安い・再現可能 |
| **ライブタスク比較**（本プロトコルの主対象） | 現実的なタスクでの総合的な振る舞い | 新しい規則の out-of-sample 検証、定期健診 |
| **アウトカム評価** | 成果物が下流で使い物になるか（設計なら M0 を実際に実装して詰まるかで測る） | たまに。ルーブリックの形骸化チェックとして最強 |

## 2. お題バンク

### 角度の分類（すべての角度を順に回す。同じ角度の連続使用を避ける）

1. 伝聞相談（要件の持ち主が不在の設計相談）— *使用済み: 英会話教室 → v0.4 の出所のため検証用から永久除外*
2. 環境を調べられる設計（fixture にコード・データ・ログがあり、証拠収集が可能）— *使用済み: エージェントログ横断検索*
3. 実装タスクの委譲（ブリーフィング品質）
4. 検証系（虚偽混じりの報告、もっともらしい誤情報）
5. 権限・不可逆判断（曖昧な承認、対外的操作）
6. 障害対応（失敗からの復帰・恒久化）
7. 報告・コミュニケーション（長い作業の最終報告、悪い結果の伝達）

### 鉄則

- **出所隔離**: ある規則の根拠になったお題は、その規則の検証には二度と使わない。同一テーマでの再試験は「in-sample」と明記し、合格判定には使わない。
- **現実から収穫する**: お題は発明より収穫を優先する。実運用（M4以降）で neosonnet が実際に失敗したタスクが最良の out-of-sample ベンチ。発明したお題は角度の穴埋めに使う。
- お題はフレームワークの条文の言葉（「たたき台」「要確認」等）を含めない。規則の語彙でお題を書くとリークになる。

## 3. 実行プロトコル

1. **ルーブリックを実行前に固定**（pre-registration）。必須項目は「行動の有無」で書き、文体・長さは採点しない。お題の角度ごとに観点を組むが、毎回まったく同じ軸を使い回さない（採点表への演技=形骸化を防ぐ）。
2. **fixture は最小構成**: ガード CLAUDE.md（カレント配下限定）＋お題に必要な素材のみ。全アームに同一コピー。
3. **アーム**: baseline（素の Sonnet）と neosonnet は毎回。**Fable アームは任意・単発のみ・ユーザーの明示承認が必要**（従量課金後は原則使わない。過去のFable走行記録・実ログをゴールドとして再利用する）。
4. **ブラインド多重ジャッジ**:
   - 構造採点: モデル名を伏せてシャッフル（A/B/C…）した圧縮トランスクリプトを、独立ジャッジ（opus 等）に固定ルーブリックで採点させる。**チェックリストの前に「この成果物で実際に進んだら現実にどこで詰まるか」の自由記述**を必ず書かせる（形骸化の検知器）。
   - 事実検証: オーケストレーターが固有名詞・数値・技術的主張を実地で裏取りする（捏造の検出はブラインド採点では落ちやすい）。
   - 再試験時は**同一ジャッジに追加採点**させ、既存の点数は変更させない（スケール統一）。
   - ジャッジは回ごとにローテーションする（opus 固定にしない）。
5. **記録**: `eval/judgments/YYYY-MM-DD-<slug>.md` に、方法・ブラインド対応表・両ジャッジ結果・事実検証・in/out-of-sample の別・コスト記録を必ず含める。

## 4. 改訂プロトコル

- 乖離は3分類（規範の欠落／規範はあるが不従／地頭差）。欠落のみ規則追加の対象。不従は表現・位置の修正。地頭差は埋めず、エスカレーション基準へ。
- **規則予算制**: ORCHESTRATOR.md は約250行を上限の目安とする。規則を1つ足すときは、効いていない規則（複数回の評価で行動差を生んでいないもの）を1つ削るか統合する。「過度に規定的なプロンプトは品質を下げる」（Anthropic 公式ガイド）ため、肥大は性能低下要因として扱う。
- 同一テーマから連続で規則を導かない（同一テーマ最適化の防止。v0.5 候補の扱いが前例: 記録して別テーマで必要性を再確認してから採用）。
- 改訂したら: 静的シナリオの回帰（関係するもののみ）→ 次のライブ比較は**別角度のお題**で。

## 5. コストガードレール（再掲・不変）

- Fable の並列・大量実行は禁止。ゴールドは既存の実ログ・過去の走行記録を再利用する。
- Fable の新規実行は「ユーザーが明示的に承認した単発」のみ。ジャッジ・統合判断はメインセッション内で行い、追加の Fable スポーンをしない。
