# アドホック比較: 曖昧で複雑な設計相談（2026-07-03）— M4 最初のデータポイント

ジャッジ: Fable 5（メインセッション）。ユーザーの発案による追試。S7（開発依頼）より複雑な「設計相談」を、素の Sonnet 5 と **インストール済みの `neosonnet` ランチャー（実運用と同一経路）** に同一条件で投げた。

## 方法

- プロンプト（曖昧・複雑を意図）: 「複数のAIエージェント（Claude Code、Codexなど）を日常的に並行で使っていて、セッションのログや成果物があちこちに散らばっています。後から『あの判断はどこでやったっけ』を探せる、横断検索・振り返りの仕組みが欲しいです。いい感じに設計を考えてもらえますか」
- 両アームとも一時ディレクトリ（ガード CLAUDE.md のみ）でヘッドレス実行。ユーザーのグローバル CLAUDE.md（委譲ルール）は両アームに載る（実運用と同条件）。
- baseline: `claude --model claude-sonnet-5` ／ framework: `~/.local/bin/neosonnet`（v0.3.1、コミット 5ef4452 時点のフレームワーク）

## 結果: 差は「賢さ」でなく「規律」に出る

| 観点 | baseline | neosonnet |
|---|---|---|
| 本題理解（判断の索引≠全文検索） | ○ | ○ 同等 |
| 設計の自己担当・実装は委譲宣言 | ○ | ○ 同等 |
| 事実と仮定の区別 | ✗ ログパス・形式を事実のように記載 | ◎ 「実在確認済み」と「一般知識ベース・実装前に要検証」をラベル分け |
| 未決事項の返し方 | △ 進め方の1問のみ | ◎ ユーザー裁量に属する4問（対象範囲・IF・バックフィル・意味検索）を選んで提示 |
| 安全・コスト観点 | ✗ なし | ◎ 機微情報の redaction 必要性、要約パスの継続トークンコスト実測、を先回り |
| 却下した案 | ✗（トレードオフ節はあり） | ◎ 3案を理由付きで却下 |
| 段階分解 | ○ Phase1-3（Phase1=実装ゼロ検証は良） | ○ M0-M3＋M0に完了条件 |
| コスト | 2ターン・99秒 | 4ターン・165秒（約1.7倍） |

## 結論

- 素の Sonnet（＋ユーザーのグローバル委譲ルール）でも設計案自体は良質。**neosonnet の価値は平均点の底上げではなく、下振れ——「もっともらしいが未検証の前提」「見えない未決事項」——の防止**にある。
- ORCHESTRATOR の反映が明確に観察できた規律: 接地（§6「主張をツール結果と突き合わせよ」→ 確認済み/要検証のラベル分け）、TBDのユーザー転記（§3.2 v0.3.1追記 → 確認事項4問）、安全の不変原則（§3.2-3 → redaction）、不採用案（§3.2-6）。
- 代償は所要時間約1.7倍・ツール呼び出し増。設計相談の文脈では妥当なコストと判断。
- headless 1ターンの限界（対話の往復なし）は前回同様。マルチターンは M4 実運用の継続観察で見る。
